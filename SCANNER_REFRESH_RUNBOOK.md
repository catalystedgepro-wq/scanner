# Scanner Refresh Runbook

## Owner

- Primary owner: `Chandrasekhar` (Scanner Ops and Refresh)

## Purpose

This runbook defines how the public Scanner site refreshes, where the schedule lives, and how to triage stale-page reports.

## Current Source of Truth

### Generation step

- The public scanner page is generated by [generate_seo_site.py](/home/operator/.openclaw/workspace/generate_seo_site.py)
- It is invoked by [run_daily_sec_catalyst.sh](/home/operator/.openclaw/workspace/run_daily_sec_catalyst.sh#L184)

### Publish target

- Public output file: [docs/index.html](/home/operator/.openclaw/workspace/docs/index.html)

### Live host scheduling

The current live host schedule is not represented by repo-local `.timer` files.

Live droplet evidence shows the scanner refresh is currently driven by `cron`:

```cron
5 8 * * 1-5 cd /opt/catalyst && bash run_daily_sec_catalyst.sh >> /var/log/catalyst.log 2>&1
```

Interpretation:

- Runs once per weekday
- Runs at `08:05 UTC`
- Equivalent to `04:05 ET` during EDT

## Findings From April 7, 2026

Live droplet checks showed:

- `/opt/catalyst/sec_catalyst_latest.csv` updated at `2026-04-07 08:06 UTC`
- `/opt/catalyst/combined_priority.csv` updated at `2026-04-07 08:07 UTC`
- `/opt/catalyst/docs/index.html` updated at `2026-04-07 08:16 UTC`
- `/var/log/catalyst.log` updated at `2026-04-07 08:16 UTC`

Conclusion:

- The scanner did refresh on April 7, 2026
- The strongest current issue is not "no refresh at all"
- The strongest current issue is likely one of:
  - schedule too sparse for expected cadence
  - stale browser/CDN caching
  - partial-refresh perception rather than generator failure

## Known Risks

- The schedule only runs once per weekday
- There is no repo-local timer file documenting the scanner cadence
- Scanner freshness expectations may exceed the actual cron cadence
- The scanner run is bundled into a larger daily script, so failures in adjacent steps can blur root cause

## First Triage Checks

When the scanner is reported stale:

1. Confirm the scheduler exists and is still active on the live host
2. Confirm [run_daily_sec_catalyst.sh](/home/operator/.openclaw/workspace/run_daily_sec_catalyst.sh) still invokes [generate_seo_site.py](/home/operator/.openclaw/workspace/generate_seo_site.py)
3. Compare live timestamps for:
   - `/opt/catalyst/sec_catalyst_latest.csv`
   - `/opt/catalyst/combined_priority.csv`
   - `/opt/catalyst/docs/index.html`
4. Check whether the issue is cadence mismatch rather than failure
5. Check whether browser/CDN caching is making the page appear older than disk

## Freshness Check Commands

Live host time:

```bash
ssh cerebro 'date -u; date'
```

Live schedule:

```bash
ssh cerebro 'crontab -l'
```

Live file timestamps:

```bash
ssh cerebro 'ls -l --time-style=long-iso /opt/catalyst/docs/index.html /opt/catalyst/combined_priority.csv /opt/catalyst/sec_catalyst_latest.csv /var/log/catalyst.log'
```

Live service health:

```bash
ssh cerebro 'systemctl status cerebro --no-pager -l | head -n 40'
```

## Likely Failure Classes

- Schedule exists, but cadence is too sparse for product expectations
- Pipeline ran, but scanner generation/publish step failed silently
- Pipeline ran successfully, but public page is cached
- Scheduler points at the right script, but environment/path drift causes partial output
- Scanner freshness is judged against intraday expectations while the actual contract is daily

## Next Recommended Change

Create an explicit scanner refresh contract:

- desired cadence
- timezone
- schedule owner
- generation owner
- publish owner
- freshness SLO for the public scanner

Without that, "not refreshing on schedule" remains ambiguous.
