What we collect — and what we don't

Two cards, side by side. The full surface, no fine print.

📥

Data we collect minimal

Everything required to bill you, authenticate you, and stop API abuse. That's it.

  • Email — for login, Stripe receipts, optional newsletter. Required.
  • Stripe customer ID — billing reference. We never see card numbers.
  • API key — bcrypt-hashed at rest. Rotatable from your account dashboard.
  • Server logs — IP, User-Agent, request path, timestamp. Used only for rate-limiting + abuse prevention.
  • Log retention — rotated & deleted after 30 days. No long-term archive.
🚫

Data we never collect never

The list of things we explicitly refuse to ingest. Some by design, some by license, all permanent.

  • Your portfolio file
  • Your brokerage / IB account
  • Your trade tape
  • Your card numbers (Stripe only)
  • Your phone or address
  • Your social handles or contact list
  • Your real name (unless you opt in for a case study)
  • Your IP location beyond country code
  • Browser fingerprints, device IDs, ad IDs

How the surface is hardened

Eight building blocks. Each one is boring on purpose — boring is what doesn't break in production.

💳

Stripe-handled billing PCI Stripe

We never see your card. Stripe Checkout is iframed; tokens come back, raw card numbers never touch our infrastructure. PCI-DSS compliance is Stripe's burden — by design.

  • Cancel anytime via the Stripe receipt link
  • Refunds processed in Stripe directly
  • Subscription state mirrored to our DB by webhook signature
  • Webhook signing secret rotated quarterly
🔑

Authentication bcrypt

Bearer-token API keys, rotatable from your account dashboard. bcrypt at rest, never logged in plaintext. Compromised? Rotate, the old key 401s instantly.

  • Format ce_live_… · 32 random bytes base32-encoded
  • Hashed (bcrypt cost 12) on creation, plaintext shown once
  • Per-key rate-limit + scope flags for /alerts vs /tape vs /dcf
  • Optional IP allowlist for Enterprise tier
🛡️

Rate-limits as DDoS defense tier-aware

Per-IP and per-API-key buckets. Bursts allowed up to 2× tier limit, then 429 with Retry-After. Persistent abuse → email warning, not a permanent ban.

  • Free: 100 req/day · Reader: 1k · Pro: 10k · Team: 50k · Enterprise: unlimited
  • Per-IP cap on unauthenticated routes to absorb scrape attempts
  • Cloudflare in front for L3/L4; nginx limit_req_zone at the edge
  • Telemetry: any 429 spike pages the founder within 60 seconds
🔐

Transport security TLS 1.3

HTTPS everywhere. No HTTP fallback for any API surface. HSTS header set; preload list submission queued. Plain-HTTP requests are 301'd to HTTPS, then HSTS-locked thereafter.

  • TLS 1.3 with modern cipher suites (no RC4, no SHA-1, no SSLv3)
  • HSTS: max-age=63072000; includeSubDomains; preload
  • Let's Encrypt cert auto-renewed every 60 days
  • SSL Labs A+ target (current grade tracked publicly via /trust audit)
👁️

No third-party tracking pixels zero

No Google Analytics. No Facebook Pixel. No Hotjar. No Sentry session replay. No LinkedIn insight tag. Just plain server logs we control. If you block cookies the site works identically.

  • One first-party cookie: tier_token for auth (HttpOnly, SameSite=Lax)
  • Zero analytics SaaS in production HTML
  • No CDNs that fingerprint visitors (we self-host fonts and CSS)
  • Dashboards built from anonymized aggregate logs only
🚪

What we don't have access to never

The boundary is one-way: we serve you data, you don't have to give us yours. We literally cannot leak what we don't store.

  • Your brokerage (IB, TD, Schwab, etc.) — no OAuth, no API key
  • Your bank — never asked, no Plaid integration
  • Your portfolio file — store it on your device, not ours
  • Your social account — no LinkedIn / Twitter OAuth
  • Your trade tape — we only see what you query us for
